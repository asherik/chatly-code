# Project Rules: Java/JavaFX Best Practices

Эти правила обязательны для любого кода в проекте. Цель: код должен быть простым, понятным, тестируемым, расширяемым и не превращаться в кашу после генерации ИИ.

## 1. Главные принципы

- Пиши простой код. Сначала читаемость, потом "умность".
- Соблюдай KISS: не усложняй архитектуру без реальной причины.
- Соблюдай DRY: не копируй одинаковую логику, выноси повторения в методы, классы, компоненты или утилиты.
- Но не доводи DRY до абсурда: если два похожих участка кода развиваются по-разному, не надо насильно склеивать их в одну абстракцию.
- Следуй SOLID, но без фанатизма. Принципы нужны для понятного кода, а не для лишних интерфейсов.
- Делай высокую связность внутри класса или модуля: класс должен заниматься одной понятной задачей.
- Делай слабую сопряженность между классами и слоями: меньше прямых зависимостей, больше явных контрактов.
- Делай разбиение по доменам и субдоменам: Project, Code Graph, Architecture, Problem Detection, Task, Agent, Workspace Safety, Runtime, LLM Gateway, UI.
- Не сваливай unrelated responsibilities в один service/manager/helper. Если модуль трудно назвать одним понятным словом, он скорее всего делает слишком много.
- Языково-специфичную логику Java/Spring/Rust/TypeScript держи в language plugins/adapters за stable SPI, а не в generic core.
- Language plugins должны быть выносимы в отдельные библиотеки без переписывания core.
- Generic core не должен импортировать Java/Spring/Rust/TypeScript plugin implementation classes.
- Код должен быть легко удалить, заменить и протестировать.
- Не пиши "магический" код, который работает только потому, что автор помнит контекст.
- Любое нетривиальное решение должно быть очевидно из названий, структуры или короткого комментария.

## 2. Языки и стек

- В проекте используется Java и JavaFX.
- Java-код должен быть современным, простым, типобезопасным и понятным.
- JavaFX используется только для UI-слоя: окна, контроллеры представлений, биндинги, события, отображение состояния.
- Для сборки исходников Chatly Code обязательно используй Gradle. Не добавляй Maven и не смешивай build tools в самом Chatly Code.
- Для coding-agent инструментов используй Spring AI Agent Utils из `C:\projects\chatly-code\examples\spring-ai-agent-utils-main` как базу. Не пиши с нуля file/shell/grep/glob/todo/skills/subagent инструменты, если библиотека уже закрывает задачу.
- Бизнес-логика не должна жить в JavaFX-контроллерах.
- Для Java-кода используй Lombok, если он уменьшает шаблонный код и не ухудшает читаемость.
- Предпочитай меньше кода там, где это не прячет важную логику.

## 3. Архитектура и слои

Слои должны быть разделены явно:

- UI/View: JavaFX FXML, views, UI-компоненты.
- Controller/ViewController: обработка UI-событий, преобразование пользовательского ввода, вызов сервисов.
- Service/Application: бизнес-сценарии, транзакции, правила приложения.
- Domain: доменные модели, бизнес-правила, value objects.
- Repository/Data Access: работа с БД, файлами, сетью или внешними источниками данных.
- DTO/ViewModel: объекты для передачи данных между слоями или отображения в UI.
- Mapper/Converter: преобразование Entity <-> DTO/ViewModel/Domain.

Правила:

- Контроллеры не должны напрямую работать с репозиториями.
- Контроллеры не должны создавать SQL-запросы, читать файлы, ходить в сеть или управлять транзакциями.
- Контроллеры не должны отдавать наружу JPA Entity или внутренние persistence-модели.
- UI-слой не должен зависеть от деталей хранения данных.
- Репозитории не должны содержать бизнес-логику.
- Entity не должны использоваться как DTO для UI или API.
- Сервисы должны работать через интерфейсы или явные зависимости, если это снижает связанность и упрощает тесты.
- Не создавай лишние слои "на всякий случай". Слой нужен, если у него есть реальная ответственность.
- Зависимости должны идти сверху вниз: UI -> Service -> Repository. Нижние слои не должны знать про верхние.

## 4. JavaFX правила

- JavaFX-контроллер должен быть тонким.
- UI должен поддерживать мультиязычность: английский по умолчанию и русский как обязательная встроенная локаль.
- Не хардкодь пользовательские строки в JavaFX-контроллерах и view.
- Все labels, buttons, tooltips, dialogs, menus и empty states должны идти через i18n/resource bundle.
- В контроллере допустимы:
  - инициализация UI;
  - биндинги;
  - обработчики кнопок, меню, таблиц и форм;
  - вызов сервисов;
  - отображение результата или ошибки.
- В контроллере запрещены:
  - бизнес-правила;
  - прямой доступ к базе данных;
  - работа с репозиториями;
  - сложные вычисления;
  - парсинг больших структур;
  - сетевые вызовы без сервиса;
  - хранение глобального состояния приложения.
- Долгие операции выполняй не в UI-thread. Используй `Task`, `Service`, virtual threads, executor или другой безопасный механизм.
- UI должен обновляться только из JavaFX Application Thread.
- Для таблиц и форм используй ViewModel/DTO, а не Entity.
- Валидацию пользовательского ввода выноси в отдельные валидаторы или сервисы, если она сложнее пары простых проверок.
- Не смешивай FXML-разметку, стили, бизнес-логику и доступ к данным в одном месте.

## 5. Modern Java правила

- Используй современные возможности Java там, где они делают код проще: records, sealed classes, pattern matching, switch expressions, text blocks.
- Используй virtual threads для большого количества блокирующих IO-задач, если это подходит сценарию.
- Используй structured concurrency для связанных фоновых операций, когда нужно управлять жизненным циклом группы задач.
- Предпочитай immutable-модели: records, final-поля, неизменяемые коллекции.
- Используй `Optional` аккуратно: для возвращаемого отсутствующего значения, но не как поле Entity/DTO без причины.
- Для ограниченных наборов состояний используй enum или sealed hierarchy.
- Для ошибок и результатов используй явные доменные типы, если простого исключения недостаточно.
- Соблюдай null-safety через JSpecify-аннотации и статический анализ, если они подключены в проекте.

## 6. Lombok правила

- Используй Lombok для уменьшения шаблонного Java-кода:
  - `@Getter`;
  - `@Setter`, только где изменяемость действительно нужна;
  - `@RequiredArgsConstructor`;
  - `@Builder`;
  - `@Value` для immutable-моделей;
  - `@Data` только для простых DTO, где безопасны equals/hashCode/toString/setters.
- Не используй `@Data` на JPA Entity без понимания последствий.
- Не включай в `toString`, `equals` и `hashCode` тяжелые связи, ленивые коллекции и циклические зависимости.
- Предпочитай constructor injection через `@RequiredArgsConstructor`.
- Не используй Lombok, если аннотация делает поведение класса менее очевидным.

## 7. Модели, Entity, DTO и Mapper

- Entity представляет структуру хранения данных, а не UI-контракт.
- DTO представляет данные для передачи между слоями или наружу.
- ViewModel представляет данные и состояние для UI.
- Domain model представляет бизнес-сущность и правила предметной области.
- Не передавай Entity в UI.
- Не возвращай Entity из сервисов, если UI или внешний код могут начать менять внутреннее состояние.
- Для преобразования используй отдельные mapper-классы или функции.
- Для повторяемого и нетривиального маппинга используй MapStruct.
- Не пиши маппинг прямо в контроллере, если он не тривиальный.
- Не добавляй в Entity методы, которые зависят от UI, репозиториев, сетевых клиентов или JavaFX.

## 8. Сервисы

- Сервис реализует пользовательский сценарий или бизнес-операцию.
- Метод сервиса должен иметь понятное имя, отражающее действие: `createUser`, `loadChatHistory`, `sendMessage`.
- Сервис не должен знать про кнопки, поля ввода, таблицы и FXML.
- Сервис должен валидировать бизнес-правила.
- Сервис должен координировать репозитории, мапперы, доменные объекты и транзакции.
- Сервис не должен быть "god object". Если сервис разросся, раздели его по сценариям или ответственности.

## 9. Репозитории и доступ к данным

- Репозиторий отвечает только за сохранение, получение, поиск и удаление данных.
- Репозиторий не должен содержать бизнес-решения.
- Названия методов должны быть точными: `findById`, `findActiveByUserId`, `save`, `deleteById`.
- Не возвращай `null`, если можно вернуть `Optional`, явный result-тип или пустую коллекцию.
- Не делай N+1 запросы. Загружай связанные данные осознанно.
- Не прячь дорогие операции за невинными названиями методов.
- Все внешние ресурсы закрывай корректно.
- Ошибки доступа к данным обрабатывай на уровне сервиса или выше, не размазывай обработку по UI-слою.

## 10. Зависимости и DI

- Используй dependency injection вместо `new` внутри бизнес-классов, если зависимость внешняя или сложная.
- Constructor injection предпочтительнее field injection.
- Класс должен явно показывать свои зависимости через конструктор.
- Не пиши самописный DI, event bus, config registry, i18n registry, ORM, migration runner, mapper framework, logging facade или metrics framework, если это уже нормально закрывается Spring/JVM-экосистемой.
- Сначала используй стандартные средства Spring Boot и Spring Framework, потом кастомный код.
- Для настроек используй typed configuration через `@ConfigurationProperties`.
- Для обычной доменной persistence используй Spring Data JPA/Hibernate.
- Для high-volume graph indexing таблиц допустим Spring JDBC/JdbcClient с prepared statements и batch transactions.
- Для миграций используй Liquibase XML. Не используй Flyway и не смешивай два подхода к миграциям в одном проекте.
- Для метрик используй Micrometer.
- Для логирования используй SLF4J/Logback, а не `System.out.println`.
- Не используй глобальное состояние без крайней необходимости.
- Singleton допустим только для stateless-сервисов или инфраструктуры, где это оправдано.
- Не создавай циклические зависимости между сервисами.

## 11. Ошибки и исключения

- Не глуши исключения пустым `catch`.
- Не возвращай `null` вместо ошибки, если ошибка важна для сценария.
- Исключения должны быть информативными, но не должны раскрывать секреты.
- Для ожидаемых бизнес-ошибок используй явные типы ошибок или checked/unchecked exceptions по стилю проекта.
- На UI показывай понятное сообщение, а не stack trace.
- Логируй технические детали там, где это помогает диагностике.

## 12. Имена и читаемость

- Имя класса должно ясно говорить, за что класс отвечает.
- Имя метода должно начинаться с действия.
- Не используй сокращения, если они не общеприняты.
- Избегай названий `Manager`, `Helper`, `Utils`, если можно назвать точнее.
- Метод должен быть коротким и делать одну вещь.
- Если метод трудно назвать, скорее всего он делает слишком много.
- Не используй "магические числа" и строки. Выноси их в константы, enum или конфигурацию.
- Условия должны читаться как бизнес-правила. Сложные условия выноси в методы с хорошими именами.

## 13. Размер классов и методов

- Класс должен иметь одну главную ответственность.
- Метод должен быть достаточно коротким, чтобы его можно было понять без прокрутки.
- Если класс знает слишком много о разных частях системы, раздели его.
- Если метод имеет много параметров, используй parameter object, DTO, builder или command object.
- Не создавай огромные util-классы.
- Не создавай абстракции ради абстракций.

## 14. Коллекции и данные

- Возвращай пустые коллекции вместо `null`.
- Предпочитай immutable collections, если данные не должны меняться.
- Не отдавай наружу изменяемые внутренние коллекции.
- Используй `List`, `Set`, `Map` как интерфейсы в сигнатурах, если конкретная реализация не важна.
- Выбирай структуру данных под задачу, а не по привычке.

## 15. Конфигурация

- Не хардкодь пути, токены, пароли, URL и настройки окружения в коде.
- Конфигурация должна быть вынесена в config-файлы, переменные окружения или настройки приложения.
- Секреты не должны попадать в git.
- Значения по умолчанию должны быть безопасными и понятными.

## 16. Тесты

- Пиши тесты для бизнес-логики, сервисов, мапперов, валидаторов и сложных условий.
- UI-тесты добавляй для важных пользовательских сценариев.
- Тест должен проверять поведение, а не внутреннюю реализацию.
- Не тестируй Lombok-generated boilerplate.
- Для сервисов мокай внешние зависимости, если это unit-тест.
- Для интеграционных тестов проверяй реальные взаимодействия между слоями.
- Название теста должно объяснять сценарий и ожидаемый результат.
- Баг без теста может вернуться. Если исправляешь баг, добавь тест, когда это разумно.

## 17. Логирование

- Логи должны помогать понять, что произошло.
- Не логируй пароли, токены, персональные данные и секреты.
- Не используй `System.out.println` для продакшен-логов.
- Используй уровни логирования осознанно:
  - `debug` для деталей диагностики;
  - `info` для важных событий;
  - `warn` для подозрительных, но не критичных ситуаций;
  - `error` для ошибок, требующих внимания.
- Не логируй одно и то же исключение много раз на разных слоях без причины.

## 18. Concurrency и async

- Не блокируй UI thread.
- Не делай shared mutable state без синхронизации или понятного ownership.
- Используй thread-safe структуры или явные механизмы синхронизации.
- Для большого количества IO-задач предпочитай virtual threads, если они подходят сценарию.
- CPU-heavy задачи вроде парсинга AST и анализа графа запускай в bounded executor, а не безлимитно.
- Multi-agent и sub-agent задачи должны иметь лимиты параллелизма, budget и понятный owner.
- Одновременно писать в один файл может только один агентный процесс через WorkspaceSafety.
- Отмена задач должна быть обработана корректно.
- Ошибки из фоновых задач должны доходить до пользователя или логов.

## 18.1 Performance правила

- Приложение должно оставаться отзывчивым: JavaFX Application Thread нельзя блокировать сканированием, LLM-вызовами, shell-командами, git, SQLite scan или graph indexing.
- Используй GraalVM как performance-направление: benchmark startup/memory/native-image, но не ломай production packaging ради преждевременной оптимизации.
- Для desktop-релиза сначала допускается JVM runtime через jpackage/install4j, если GraalVM native-image создает хрупкость.
- Индексация проекта должна быть incremental: content hash, skip unchanged files, batch writes, transactions.
- Длинные операции должны иметь progress, cancellation и понятный статус.
- Кэши должны быть ограничены по размеру.
- Логи и agent events не должны бесконечно расти в UI.

## 19. Безопасность

- Валидируй входные данные.
- Не доверяй данным из UI, файлов, сети или БД без проверки.
- Не собирай SQL или команды через конкатенацию строк.
- Используй prepared statements, ORM-параметры или безопасные API.
- Не храни секреты в коде.
- Не показывай пользователю внутренние технические ошибки.
- Ограничивай права доступа там, где есть роли или учетные записи.

## 20. Производительность

- Не оптимизируй преждевременно.
- Сначала сделай код понятным и корректным, потом измеряй узкие места.
- Не выполняй тяжелые операции в цикле, если можно сделать batch.
- Не загружай большие данные полностью в память без необходимости.
- Используй lazy loading, pagination или streaming, если данных много.
- Не делай дорогую работу в JavaFX binding/listener без контроля частоты вызовов.

## 21. Комментарии и документация

- Комментарии нужны, чтобы объяснить "почему", а не пересказать "что".
- Не оставляй устаревшие комментарии.
- Публичные API и сложные доменные правила документируй.
- TODO должен содержать конкретное действие, а не расплывчатую мысль.
- Не оставляй мертвый код в комментариях.

## 22. Форматирование и стиль

- Следуй единому стилю форматирования проекта.
- Не смешивай разные стили в одном файле.
- Используй автоформаттер.
- Импорты должны быть чистыми, без неиспользуемых зависимостей.
- Один файл должен содержать одну основную public-сущность.
- Не делай огромные строки.
- Код должен проходить статический анализ, если он настроен.

## 23. Правила для ИИ-генерации кода

- Не генерируй код в контроллеры, если логика относится к сервисам, домену или репозиториям.
- Не используй Entity как универсальный объект для всего.
- Перед добавлением нового класса проверь, нет ли уже подходящего класса или паттерна в проекте.
- Сохраняй существующую архитектуру и стиль проекта.
- Не добавляй новую библиотеку без причины.
- Не создавай абстракцию, если есть только одна реализация и нет понятной выгоды.
- Не пиши большие куски кода без разделения по ответственности.
- Не меняй публичные контракты без необходимости.
- Не ломай обратную совместимость без явного решения.
- Если есть выбор между коротким понятным кодом и длинным "идеальным" паттерном, выбирай короткий понятный код.
- Если можно написать меньше кода без потери смысла, пиши меньше кода.
- После изменения кода проверь компиляцию и тесты, если это возможно.

## 24. Definition of Done

Код считается готовым, если:

- ответственность классов понятна;
- слои не смешаны;
- контроллеры тонкие;
- бизнес-логика находится в сервисах или домене;
- репозитории отвечают только за доступ к данным;
- Entity не торчат в UI;
